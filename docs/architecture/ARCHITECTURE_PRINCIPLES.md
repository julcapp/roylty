# ARCHITECTURE_PRINCIPLES

Статус: Approved
Версия: 1.0
Проект: Soft ICE Platform / «У Тимоши»

## Назначение

Документ фиксирует базовые архитектурные принципы разработки Soft ICE Platform.

## 1. GitHub — единственный источник истины

Все изменения должны фиксироваться в репозитории. Не должно быть файлов, которые существуют только на сервере, локальном компьютере или в переписке.

## 2. Документация создаётся вместе с кодом

Каждая значимая подсистема сопровождается документацией, тестовыми сценариями и, при необходимости, записью в архитектурном журнале.

## 3. Данные отделены от интерфейса

React-компоненты не должны хранить цены, пути к изображениям, списки товаров и бизнес-правила. Эти данные должны поступать через каталог и сервисы.

## 4. Один каталог для всех платформ

Единый каталог продукции используется в Mini App, Web App, Telegram Bot, MAX Mini App, CRM, Seller App и интерфейсе вендингового автомата.

## 5. Адаптивность по умолчанию

Каждый экран проектируется для мобильных устройств, планшетов, desktop, терминалов и больших сенсорных экранов.

## 6. Повторное использование компонентов

Компоненты должны проектироваться так, чтобы их можно было использовать повторно в разных экранах и каналах.

## 7. Медиатека является отдельной подсистемой

Изображения продуктов, сиропов, топпингов, бренда, Тимофея, маркетинга и UI хранятся централизованно и имеют метаданные.

## 8. Бизнес-логика находится в сервисах

Расчёт цены, выбор изображения, конфигурация заказа, скидки и правила лояльности реализуются в отдельных сервисах, а не внутри экранов.

## 9. Новые функции вводятся безопасно

Для крупных или рискованных возможностей используются Feature Flags, тестовые сценарии и поэтапные релизы.

## 10. Версионирование обязательно

Каждый завершённый этап сопровождается обновлением CHANGELOG, проверкой, синхронизацией GitHub/GitHub Desktop и Git Tag.

## 11. Сначала модель — потом интерфейс

Перед реализацией нового модуля сначала описываются доменная модель, структура данных, бизнес-правила и сценарии использования.

## 12. Инкрементная разработка

Работа ведётся законченными инженерными инкрементами: архитектура, код, тесты, документация, фиксация версии.
